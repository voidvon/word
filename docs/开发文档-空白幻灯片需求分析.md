## 2. 需求总览

这是一个围绕“英文网页浏览辅助 + 查词 + 单词管理 + 单词本学习”的产品方案。当前 PPT 和原型主要描述的是 App 内的左侧边栏部分，不是一个孤立的整屏词典应用。

完整产品形态需要按两栏理解：

1. 左栏：当前项目正在实现的词典、搜索历史、每日推荐、单词块、知识圈等功能模块
2. 右栏：用户正在浏览的英文网页或其它内容界面
3. 跨栏通道：右侧网页通过 App 内置通道读取左栏用户数据，并把浏览、选词、点击等信息回传给左栏
4. 注入能力：后续通过 JS 注入到右侧网页中，实现类似 Relingo 的单词提示，并逐步扩展更多网页增强功能

因此，当前 MVP 既要能作为左栏独立运行，也要为后续“右侧网页读取状态、回传行为、注入提示”预留数据和接口边界。

当前核心由以下 5 部分组成：

1. 首页 `/index`
2. 查词详情页 `/?wd=<word>&l=en`
3. 单词块首页 `/wdBook`
4. 单词本详情页 `/wdBook`
5. 全局单词数据与用户学习状态数据

当前版本的重点不是把整套产品完整做完，而是先实现一个可运行的 MVP，覆盖：

1. 首页搜索与历史记录展示
2. 查词详情页基础信息展示
3. 单词块首页的概览视图
4. 我的单词本列表
5. 单词本详情页的单词网格展示
6. 基础本地数据结构与用户状态维护
7. 为右侧网页注入和跨栏通信保留用户状态读取、单词状态查询、行为回传入口

以下内容在 PPT 中明确提到“暂时不写”或“未来可加”：

1. 首页左上角“三”字按钮的完整功能
2. 单词块首页顶部黑色统计模块的完整数据接入
3. “今日新词”模块及其逻辑；该模块后续删除
4. “重点关注”“常查单词”“不认识”三个汇总模块的完整页面逻辑；首版只保留数据语义
5. OCR/文件识别导入
6. 单词册功能
7. 左上角菜单中除排序切换外的其它功能
8. 右侧网页的完整 JS 注入脚本
9. 类 Relingo 的网页内单词高亮、释义浮层和学习状态标记
10. 跨栏通信的生产级权限、隔离和同步机制

## 4. 页面与功能需求

## 4.1 首页 `index`

### 4.1.1 页面目标

首页承担左侧边栏中的统一查词入口，同时承担搜索历史展示。每日推荐、单词块、知识圈作为独立模块放在底部导航中，不再作为搜索框下方的 Tab。

从完整 App 视角看，首页属于左栏。用户在右侧网页浏览英文内容时，右侧页面可以通过 App 内置通道读取左栏维护的词典和学习状态，并将用户在网页中的选词、悬停、查询、加入学习等行为回传到左栏状态。

### 4.1.2 页面结构

1. 顶部搜索框
2. 搜索框左侧菜单按钮
3. 搜索框右侧语音/拍照入口
4. 页面主内容区
5. 底部导航栏

### 4.1.3 交互要求

1. 点击搜索框后，出现查询历史列表
2. 历史列表展示 `searchList` 对应内容
3. 点击历史词条可进入该词查词页
4. 点击底部导航“每日推荐 / 单词块 / 知识圈”进入对应独立模块

当前不做：

1. 左上角菜单按钮交互
2. 每日推荐区完整排版
3. 右侧网页注入脚本的完整交互

### 4.1.4 数据要求

首页依赖本地维护的 `searchList`。

PPT 中对 `searchList` 有两处表述：

1. 一处写“需要去重合并，并统计查询次数”
2. 另一处写“数组里只需要单词即可，无需其他数据，词性词义去 `wordJson` / `wordJsonU` 获取”

结合这两处信息，更合理的实现建议是：

1. `searchList` 只存单词顺序列表，按最近查询时间倒序，去重
2. 查询次数不要存在 `searchList` 数组本身
3. 查询次数存到 `wordJsonU[word].sc`

这样可以同时满足“展示历史”和“统计查询次数”，并避免重复数据。

### 4.1.5 展示字段建议

历史列表每项建议展示：

1. 单词
2. 简短释义
3. 查询次数或最近查询痕迹

其中释义不在 `searchList` 中冗余存储，运行时从 `wordJson` 读取。

### 4.1.6 与右侧网页的关系

首页和其它左栏模块维护的数据，后续需要暴露给右侧网页注入脚本使用。首版可先把数据落在 LocalStorage 等本地状态中，后续再抽象成 App 内置桥接方法。

建议预留的跨栏能力包括：

1. 右侧网页查询某个单词是否在词典中
2. 右侧网页查询某个单词的学习状态、是否忽略、是否已掌握
3. 右侧网页把用户在网页中的查词行为写入 `searchList`
4. 右侧网页把用户加入学习、忽略、标记掌握等操作回传到左栏状态
5. 左栏状态变更后通知右侧网页刷新高亮、提示和浮层内容

## 4.2 查词详情页 `/?wd=<word>&l=en`

### 4.2.1 页面目标

展示单词基础信息，并成为单词加入学习流的入口。

### 4.2.2 页面结构

当前根据 PPT，仅明确要求完成头部基础信息：

1. 单词
2. 英/美音或至少一个音标展示
3. 基础词义与词性

图片中虽然存在更多区块，但 PPT 文本只要求头部两行参考欧路词典，其余内容未明确要求复刻，因此当前版本只建议实现基础详情结构，不建议把整张参考图全部照搬。

### 4.2.3 路由规则

路由示例为：

`/?wd=evidence&l=en`

说明页面至少接收：

1. `wd`：单词
2. `l`：语言参数

### 4.2.4 数据来源

主要依赖 `wordJson[word]`：

1. 音标
2. 词性
3. 释义
4. 查词页界面标注需要的基础数据

### 4.2.5 与用户状态的关系

查询单词时需要同步更新：

1. `searchList`
2. `wordJsonU[word].sc`
3. `wordJsonU[word].t` 可按需要记录最近查询时间

## 4.3 单词块首页 `/wdBook`

### 4.3.1 页面目标

这是所有用户导入单词与学习状态的总入口，分为两部分：

1. AI 单词本分类概览
2. 我的单词本管理

### 4.3.2 页面结构

根据截图，页面可拆为：

1. 顶部黑色统计模块
2. `AI单词本` 分类区
3. `我的单词本` 卡片区

### 4.3.3 顶部统计模块

当前只做样式占位，不对接真实统计数据。

可先保留如下展示位：

1. 总词汇量
2. 已掌握比例
3. 今日学习次数
4. 今日学习时长
5. 连续打卡信息

但这些字段在本期都不作为真实验收项。

### 4.3.4 AI 单词本分类

PPT 说明此区域是“所有导入单词的整体概览”。

最新分类语义如下：

1. 遗忘中的单词：核心功能。来源为全部 `s="a"` 的单词，但列表只显示 `t <= 当前时间戳` 的到期项；`t > 当前时间戳` 表示仍在记忆冷却期。
2. 今日新词：不再实现逻辑，模块后续删除。
3. 易错词：`s="a" && ec >= 3` 的单词集合；`ec` 为点击“忘记”的累计错误次数。
4. 不认识：全部 `s="n"` 的新导入单词，即已进入单词本但从未进行背诵操作的单词。
5. 重点关注：等同于收藏，由单词界面的“重点关注”按钮控制。
6. 砍掉的单词：全部 `s="c"` 的单词，表示用户主动判定已经掌握。
7. 常查单词：查询次数 `sc >= 3` 时自动进入，阈值 `N` 暂定为 `3`。
8. 忽略的单词：全部 `s="b"` 的单词，不参与正常学习和背诵。

实现优先级：

1. “遗忘中的单词”及其冷却时间判断是核心，必须正确实现。
2. “重点关注”“常查单词”“不认识”暂不要求完成汇总模块 UI，但数据模型和派生规则必须保留。
3. “易错词”“砍掉的单词”“忽略的单词”按状态或计数派生，不重复存储单词详情。

### 4.3.5 我的单词本

该区域复刻“单词块”的单词本管理逻辑，至少包括：

1. 单词本列表展示
2. 创建单词本
3. 查看单词本详情
4. 单词本增删改查

未来能力但本期不做：

1. 从照片/文件中识别导入
2. 单词册功能

### 4.3.6 卡片展示建议

根据截图，单词本卡片应至少包含：

1. 单词本名称
2. 单词数量
3. 视觉色块/进度色带
4. 更多操作入口

PPT 指出颜色会受“总词数 / 已掌握词数”的比值影响，因此卡片颜色不是纯静态皮肤，后续应支持根据进度映射。

## 4.4 单词本详情页 `/wdBook`

### 4.4.1 页面目标

展示某个单词本内的单词网格，并支持按不同规则排序浏览。

### 4.4.2 页面结构

从截图可见：

1. 顶部栏显示当前单词本名称
2. 顶部提供搜索入口
3. 主体为单词方块矩阵

### 4.4.3 排序要求

必须预留两种排序方式：

1. 按添加顺序
2. 按字母顺序

PPT 指出参考 App 默认是按字母排序，但本产品要额外支持按添加顺序。

### 4.4.4 交互要求

1. 排序模式在左上角功能按钮中切换
2. 左上角按钮弹出的其他功能暂不实现

### 4.4.5 数据要求

每个 `book` 需要维持两个顺序视图：

1. 添加顺序数组
2. 字母顺序数组

单词本单词上限为 2000 个。

### 4.4.6 颜色语义

单词方块在正常背诵状态 `s="a"` 时，按阶段 `a=0~7` 显示：

1. `a=0`：灰色
2. `a=1`：红色
3. `a=2`：橙色
4. `a=3`：黄色
5. `a=4`：绿色
6. `a=5`：青色
7. `a=6`：蓝色
8. `a=7`：紫色

阶段色与状态色统一配置化。暂定非背诵状态颜色：`n` 蓝灰色 `#78909C`、`b` 深灰色 `#455A64`、`c` 品红色 `#D81B60`、`d` 金色 `#B8860B`。

## 5. 数据模型需求

## 5.1 全局词典数据 `wordJson`

PPT 定义：

```json
{
  "go": {},
  "advance": {}
}
```

该结构本质是一个按单词 key 索引的总词库，负责存储基础词典信息。

应包含但不限于：

1. 音标
2. 词性
3. 释义
4. 查词页与页面标注所需的其它基础数据

数据来源：

1. 从服务器获取
2. 按需加载

说明：当前仓库 `src/data/dictionary/*.json` 很像单词级静态词典数据，后续可以评估是否作为 `wordJson` 的本地数据来源。

## 5.2 用户状态数据 `wordJsonU`

最新需求定义：

```json
{
  "go": {
    "s": "n/a/b/c/d",
    "a": 2,
    "sc": 5,
    "t": 1761955200000,
    "l": [1, 3],
    "ec": 1,
    "previousStatus": null,
    "displayText": "Go"
  },
  "advance": {}
}
```

推荐按单词为 key 保存，示意如下：

```json
{
  "go": {
    "s": "a",
    "a": 2,
    "sc": 5,
    "t": 1761955200000,
    "l": [1, 3],
    "ec": 1
  }
}
```

字段说明：

1. `s`：单词状态
2. `a`：艾宾浩斯学习阶段 `0~7`；新词状态 `n` 默认没有该字段，首次背诵后创建
3. `sc`：查询次数
4. `t`：下一次允许背诵的时间戳；在 `s=b/c` 时保留原值，不重新计算
5. `l`：所属单词本 id 列表
6. `ec`：错误次数，每次点击“忘记”增加 `1`
7. `previousStatus`：进入 `b/c` 前的原状态，仅允许 `n/a/null`，用于取消忽略或砍掉后恢复
8. `displayText`：可选的原始展示文本；数据 key 使用规范化值，展示时保留用户原始大小写

任何用户导入到单词本中的文本都必须写入 `wordJsonU`。导入内容不限英语，允许任何文字；基础词典中没有对应词条也不能阻止导入。

### 5.2.1 状态字段 `s`

最新定义：

1. `n`：new，新添加到单词本但从未背诵
2. `a`：正常背诵流程
3. `b`：ban，表示忽略，不进行学习和背诵
4. `c`：cut，表示砍掉，即用户判定已经掌握
5. `d`：ed，表示已背会

说明：

1. 新导入单词默认 `s="n"`，归入不认识集合 `nList`。
2. 首次执行“认识 / 模糊 / 忘记”任一背诵操作后，`s` 改为 `a` 并进入学习集合 `aList`。
3. `b` 对应忽略集合 `bList`，`c` 对应砍掉集合 `cList`。
4. 当阶段已经是 `a=7` 时再次点击“认识”，将 `s` 改为 `d`，表示完成背诵。

### 5.2.2 学习阶段字段 `a`

`a` 取值 `0~7`，对应间隔为：

1. `0`：首次进入背诵流程
2. `1`：20 分钟后
3. `2`：1 小时后
4. `3`：9 小时后
5. `4`：1 天后
6. `5`：2 天后
7. `6`：6 天后
8. `7`：31 天后

### 5.2.3 时间戳字段 `t`

最新需求特别强调 `t` 逻辑最复杂。

规则应为：

1. 当执行“认识 / 模糊 / 忘记”时，先调整阶段 `a`，再令 `t = 当前时间戳 + 新阶段对应时间间隔`。
2. `t > 当前时间戳` 表示仍在记忆冷却期：遗忘列表隐藏该词，背诵卡隐藏“认识 / 模糊 / 忘记”按钮，并提示“该词在记忆冷却期中”。
3. `t <= 当前时间戳` 表示已到复习时间，应在遗忘列表中显示并允许背诵。
4. 执行忽略 `b` 或砍掉 `c` 时，不修改已有的 `a` 和 `t`，以便取消操作后恢复原背诵进度。

需求原始说明中有一句“`t` 小于当前时间说明不需要背诵”，与“隐藏 `t > 当前时间`”及冷却期定义冲突。本文按后两项一致语义记录：`t <= 当前时间` 为到期可背。

## 5.3 搜索历史 `searchList`

PPT 定义：

```json
["word0", "word1"]
```

规则：

1. 按时间从新到旧
2. 去重

建议：

1. 仅保存单词顺序
2. 查询次数放到 `wordJsonU[word].sc`
3. 展示时再从 `wordJson` 读取释义

## 5.4 状态派生列表

以下列表均以 `wordJsonU` 为权威数据源：

1. `nList`：`s="n"`，新添加但从未背诵的单词。
2. `aList`：`s="a"`，处于正常背诵流程的全部单词。
3. 遗忘列表可见项：`aList` 中满足 `t <= 当前时间戳` 的单词。
4. 冷却项：`aList` 中满足 `t > 当前时间戳` 的单词，不在遗忘列表显示。
5. `bList`：`s="b"`，已忽略的单词。
6. `cList`：`s="c"`，已砍掉的单词。
7. `eList`：`s="a" && ec >= 3` 的易错词。

推荐由 `wordJsonU` 动态派生这些列表，避免同时维护多份永久数组造成状态错位。如果后续为性能增加物化索引，必须保证状态变更时原子地从旧列表移除并加入新列表。

## 5.5 单词本索引 `wdBookList`

PPT 定义：

```json
[id0, id1, id2]
```

规则：

1. id 可使用数字自增
2. 主要用于排序展示
3. 数组长度通常不会超过 20，极限不超过 100

## 5.6 单词本详情字典 `wdBookJson`

PPT 给了两类对象：

1. `book`
2. `list`

建议结构如下：

```json
{
  "1": {
    "s": "b",
    "n": "雅思核心词",
    "r": [50, 13],
    "wordsByAdd": ["go", "advance"],
    "wordsByAlpha": ["advance", "go"],
    "articleWordCounts": { "go": 12, "advance": 3 }
  },
  "2": {
    "s": "l",
    "n": "英语课本",
    "l": [1, 3, 5]
  }
}
```

字段含义：

1. `s=b`：book
2. `s=l`：list，表示单词册/单词本集合
3. `n`：名称
4. `r:[total, mastered]`：总词数和已掌握词数
5. `l`：该集合下包含的 book id

PPT 原文没有给出 `book` 内部保存单词数组的字段名，但由于第 5 页明确要求两种排序都要支持，因此实现上建议补充：

1. `wordsByAdd`
2. `wordsByAlpha`
3. `articleWordCounts`：按单词本永久保存的文章出现次数；重复文章导入时累加

否则无法稳定支撑排序切换。

## 5.7 变更数据集 `upDateList`

PPT 提到了：

```json
[]
```

名称为 `upDateList`，说明是“变更数据集”。

PPT 没有解释其最终用途，但结合场景，可能是：

1. 用户状态变更待同步队列
2. 本地数据脏标记集合
3. 学习状态批量更新日志

当前建议：

1. 先预留结构
2. 不作为首版必须实现项
3. 若有离线同步需求，再细化其数据格式

## 6. 业务规则整理

## 6.1 搜索行为

当用户查询一个单词时，至少应执行：

1. 若 `wordJson` 中无该词，触发远程加载
2. 更新 `searchList`，置顶并去重
3. `wordJsonU[word].sc + 1`
4. 记录最近查询相关时间

## 6.2 单词导入

任何文本导入单词本时：

1. 不限制语言和字符集，词典中不存在的文本也允许导入。
2. 若 `wordJsonU[text]` 不存在，则创建记录并令 `s="n"`、`sc=0`、`ec=0`、`l=[bookId]`。
3. 新状态 `n` 不创建背诵阶段 `a`；`t` 是否在新词阶段存在暂不强制。
4. 若记录已存在，只把当前单词本 id 去重加入 `l`，不得覆盖已有学习状态、阶段、时间和计数。
5. 同步更新所属 `book` 的单词数组。

## 6.3 正常背诵操作

“认识 / 模糊 / 忘记”三种操作都会令 `s="a"`。先读取原阶段；若不存在 `a`，按 `0` 作为首次背诵阶段，再执行：

1. 认识：`a = min(7, a + 1)`。
2. 模糊：`a` 不变。
3. 忘记：`a = max(0, a - 1)`，并令 `ec += 1`。
4. 三种操作最后统一计算 `t = 当前时间戳 + 阶段 a 对应时间间隔`。
5. 状态从 `n` 首次转为 `a` 时，从 `nList` 移除并加入 `aList`。
6. 当前阶段已经是 `7` 时再次点击认识，不再计算下一阶段，直接令 `s="d"`。

## 6.4 遗忘列表与记忆冷却期

遗忘列表是核心背诵入口：

1. 候选集合为全部 `s="a"` 的单词。
2. 只显示 `t <= 当前时间戳` 的到期单词。
3. `t > 当前时间戳` 的单词处于冷却期，不在遗忘列表显示。
4. 若用户通过其它入口打开冷却期单词的背诵卡，隐藏“认识 / 模糊 / 忘记”按钮，显示“该词在记忆冷却期中”。

## 6.5 忽略与砍掉

当用户执行忽略或砍掉时：

1. 忽略：先把原 `s` 写入 `previousStatus`，再把 `s` 改为 `b`。
2. 砍掉：先把原 `s` 写入 `previousStatus`，再把 `s` 改为 `c`。
3. 根据原状态从 `nList` 或 `aList` 移除。
4. 不修改原有的 `a` 和 `t`。
5. 取消忽略或取消砍掉时，把 `s` 恢复为 `previousStatus`，然后清空 `previousStatus`；原 `a/t` 立即恢复生效。

## 6.6 其它派生模块

1. 易错词：每次点击“忘记”令 `ec += 1`，`s="a" && ec >= 3` 时进入 `eList`。
2. 常查单词：每次查询令 `sc += 1`，`sc >= 3` 时进入常查集合。
3. 重点关注：等同于收藏，由单词页面按钮切换。
4. 不认识：`s="n"` 的单词集合。
5. 今日新词：不实现，模块后续删除。

上述集合优先动态派生。重点关注、常查单词和不认识三个汇总模块暂不要求实现页面逻辑。

## 6.7 数据 key 与原始文本

英文直接全部转小写会丢失 `US/us`、`Polish/polish` 等可能有意义的大小写差异，因此采用双轨方案：

1. `wordJsonU` 的 key 用于查找和默认去重：执行 Unicode `NFKC` 规范化、去除首尾空白，并对拉丁字母转小写。
2. `displayText` 保存首次导入或用户确认的原始文本，用于界面展示。
3. 中文及其它无大小写文字保持原字符，仅执行 Unicode 规范化和首尾空白清理。
4. 若未来需要把仅大小写不同的词作为独立词条，再为对应导入入口增加“区分大小写”选项。

仅查询但未加入单词本的内容仍写入 `wordJsonU`，但允许暂时没有 `s`：例如 `{ sc: 1, ec: 0, l: [] }`。加入任一单词本时再补写 `s="n"` 和 book id。这样继续满足 `sc` 位于 `wordJsonU` 的原模型，也不会让查询词误入不认识列表。

## 6.8 从文章中导入（核心功能完成后开发）

单词本新增第三种添加方式“从文章中导入”：

1. 打开多行输入框，允许用户直接粘贴文章。
2. 首版只处理英文；调用全局文本分词工具提取 token，标点和空白作为分隔符，不能简单删除后把相邻单词拼接。
3. 按规范化 key 去重并统计每个 token 的出现次数。
4. 默认按出现次数降序排列；次数相同时按首次出现顺序排列。
5. 撇号和连字符属于英文 token 的内部字符，`don't`、`state-of-the-art` 分别作为一个词。
6. 数字、邮箱和 URL 不作为可导入单词。
7. 用户确认后按该顺序批量加入目标单词本。
8. 出现次数按单词本永久保存；同一词再次从文章导入时累加原次数。
9. 分词函数放在全局工具目录，供单词本导入、网页插件和其它页面共同调用。

推荐工具接口：

```ts
type TokenFrequency = {
  key: string
  displayText: string
  count: number
  firstIndex: number
}

function extractTokenFrequencies(text: string, locale?: string): TokenFrequency[]
```

首版使用独立、可测试的英文 tokenizer 实现上述 token 规则，不引入网络依赖。未来扩展到无空格语言时，再在同一工具接口下接入 `Intl.Segmenter` 或专业分词库。

## 8. 技术实现建议

## 8.1 数据分层

建议分成三层：

1. 词典基础层：`wordJson`
2. 用户状态层：`wordJsonU`
3. 视图索引层：`searchList`、由状态派生的 `nList/aList/bList/cList/eList`、`wdBookList`、`wdBookJson`

这样做的好处是：

1. 基础词典数据与用户行为解耦
2. 视图排序可独立优化
3. 后续远程同步或本地缓存更容易扩展

## 8.2 不要重复存释义

PPT 已明确表达“历史数组只需要单词即可”，因此：

1. `searchList`
2. 状态派生列表或其物化索引
3. `book` 内单词数组

都尽量只存单词 key，不重复存释义、音标等详情。

## 8.3 排序能力要在数据层准备好

因为单词本详情页必须支持两种排序，所以建议不要在每次渲染时都临时排序大数组，而是预先维护：

1. `wordsByAdd`
2. `wordsByAlpha`

这样切换排序只是切换数据源引用。

## 8.4 颜色映射配置化

正常背诵状态的阶段色已经明确，建议设计：

```ts
const reviewStageColors = {

  1: "red",
  2: "orange",
  3: "yellow",
  4: "green",
  5: "cyan",
```

暂定状态色同样放入配置：`n=#78909C`、`b=#455A64`、`c=#D81B60`、`d=#B8860B`。

阶段间隔、阶段颜色、状态颜色和阈值集中放入 `src/config/review.ts`，业务代码不得散落硬编码：

## 11. 结论

这份 PPT 的本质不是单一页面需求，而是一套“词典查询 + 生词管理 + 单词本学习”的产品草案。

从开发角度，当前最重要的是先把以下三件事做对：

1. 页面范围收敛，只做 MVP
2. 数据结构先定清楚，尤其 `wordJsonU` 的 `n/a/b/c/d` 状态、阶段时间和派生列表
3. 把“当前必做”和“未来预留”严格分开

只要先完成这三点，后续无论是接真实词典数据、补统计模块，还是扩展 OCR 和单词册，都不会推翻首版结构。
