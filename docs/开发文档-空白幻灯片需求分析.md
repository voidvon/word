# 空白幻灯片需求分析与开发文档

## 1. 文档说明

本文件基于 `空白幻灯片.pptx` 的 6 页内容、1 页备注以及 4 张内嵌界面示意图整理。

目标是把 PPT 中偏零散的想法，收敛成一份可用于开发排期、页面实现和数据建模的文档。

## 2. 需求总览

这是一个围绕“查词 + 单词管理 + 单词本学习”的词典/背词产品方案，核心由以下 5 部分组成：

1. 首页 `/index`
2. 查词详情页 `/?wd=<word>&l=en`
3. 单词块首页 `/wdBook`
4. 单词本详情页 `/wdBook`
5. 全局单词数据与用户学习状态数据

当前版本的重点不是把整套产品完整做完，而是先实现一个可运行的 MVP，覆盖：

1. 首页搜索与历史记录展示
2. 查词详情页基础信息展示
3. 单词块首页的概览视图
4. 我的单词本列表
5. 单词本详情页的单词网格展示
6. 基础本地数据结构与用户状态维护

以下内容在 PPT 中明确提到“暂时不写”或“未来可加”：

1. 底部导航栏交互
2. 首页左上角“三”字按钮的完整功能
3. 单词块首页顶部黑色统计模块的完整数据接入
4. “常查的单词”与“忽略的单词”两个新增分类
5. OCR/文件识别导入
6. 单词册功能
7. 左上角菜单中除排序切换外的其它功能

## 3. 视觉与参考来源

PPT 中的图片给出了明确的参考方向：

### 3.1 首页参考

首页搜索区、历史记录弹层和整体风格参考手机 App“欧路词典”。

从图片可见：

1. 顶部有搜索框
2. 搜索框左侧有菜单入口，右侧有语音/拍照入口
3. 搜索框下方有 3 个 Tab
4. 搜索框聚焦后，下方展示历史查询列表
5. 历史项至少包含单词、简短释义、历史查询标识

需要注意：图片上的 Tab 文案是“每日一句 / 每日听力 / 知识圈”，但 PPT 文字要求改为“每日推荐 / 单词块 / 知识圈”。开发时应以 PPT 文字要求为准。

### 3.2 查词详情页参考

详情页参考欧路词典头部两行布局，只保留：

1. 单词本身
2. 音标/读音

明确不需要：

1. 全球发音
2. 跟读功能

### 3.3 单词块首页参考

整体参考“单词块”App 首页：

1. 上方统计区
2. AI 单词本分类卡片区
3. 我的单词本卡片区

但顶部黑色统计模块当前只需要做大概样式，占位即可，不需要真实接数据。

### 3.4 单词本详情页参考

单词本详情页参考“单词块”App 的词块网格页：

1. 顶部有标题
2. 搜索按钮存在
3. 内容主体是单词色块网格
4. 需要支持不同颜色表达不同状态

备注页明确给出部分颜色语义：

1. 灰色表示忽略
2. 橙色表示应该学习
3. 紫色也有特殊语义，但 PPT 备注未写完整，当前定义不充分

因此紫色相关规则应先保留配置位，等待产品补充。

## 4. 页面与功能需求

## 4.1 首页 `index`

### 4.1.1 页面目标

首页承担统一查词入口，同时承担搜索历史展示，并提供进入“单词块”模块的入口。

### 4.1.2 页面结构

1. 顶部搜索框
2. 搜索框左侧菜单按钮
3. 搜索框右侧语音/拍照入口
4. 搜索框下方 3 个 Tab
5. 页面主内容区

Tab 为：

1. 每日推荐
2. 单词块
3. 知识圈

### 4.1.3 交互要求

1. 点击搜索框后，出现查询历史列表
2. 历史列表展示 `searchList` 对应内容
3. 点击历史词条可进入该词查词页
4. 点击“单词块”Tab 后进入单词块相关内容或跳转到对应模块

当前不做：

1. 底部导航栏交互
2. 左上角菜单按钮交互
3. 每日推荐区完整排版

### 4.1.4 数据要求

首页依赖本地维护的 `searchList`。

PPT 中对 `searchList` 有两处表述：

1. 一处写“需要去重合并，并统计查询次数”
2. 另一处写“数组里只需要单词即可，无需其他数据，词性词义去 `wordJson` / `wordJsonU` 获取”

结合这两处信息，更合理的实现建议是：

1. `searchList` 只存单词顺序列表，按最近查询时间倒序，去重
2. 查询次数不要存在 `searchList` 数组本身
3. 查询次数存到 `wordJsonU[word].sc`

这样可以同时满足“展示历史”和“统计查询次数”，并避免重复数据。

### 4.1.5 展示字段建议

历史列表每项建议展示：

1. 单词
2. 简短释义
3. 查询次数或最近查询痕迹

其中释义不在 `searchList` 中冗余存储，运行时从 `wordJson` 读取。

## 4.2 查词详情页 `/?wd=<word>&l=en`

### 4.2.1 页面目标

展示单词基础信息，并成为单词加入学习流的入口。

### 4.2.2 页面结构

当前根据 PPT，仅明确要求完成头部基础信息：

1. 单词
2. 英/美音或至少一个音标展示
3. 基础词义与词性

图片中虽然存在更多区块，但 PPT 文本只要求头部两行参考欧路词典，其余内容未明确要求复刻，因此当前版本只建议实现基础详情结构，不建议把整张参考图全部照搬。

### 4.2.3 路由规则

路由示例为：

`/?wd=evidence&l=en`

说明页面至少接收：

1. `wd`：单词
2. `l`：语言参数

### 4.2.4 数据来源

主要依赖 `wordJson[word]`：

1. 音标
2. 词性
3. 释义
4. 查词页界面标注需要的基础数据

### 4.2.5 与用户状态的关系

查询单词时需要同步更新：

1. `searchList`
2. `wordJsonU[word].sc`
3. `wordJsonU[word].t` 可按需要记录最近查询时间

## 4.3 单词块首页 `/wdBook`

### 4.3.1 页面目标

这是所有用户导入单词与学习状态的总入口，分为两部分：

1. AI 单词本分类概览
2. 我的单词本管理

### 4.3.2 页面结构

根据截图，页面可拆为：

1. 顶部黑色统计模块
2. `AI单词本` 分类区
3. `我的单词本` 卡片区

### 4.3.3 顶部统计模块

当前只做样式占位，不对接真实统计数据。

可先保留如下展示位：

1. 总词汇量
2. 已掌握比例
3. 今日学习次数
4. 今日学习时长
5. 连续打卡信息

但这些字段在本期都不作为真实验收项。

### 4.3.4 AI 单词本分类

PPT 说明此区域是“所有导入单词的整体概览”。

当前保留单词块原有 6 种分类，未来预留 2 种：

1. 遗忘中的单词，改名为“待复习单词”
2. 今日新词
3. 易错词
4. 不认识
5. 重点关注
6. 砍掉的单词
7. 常查的单词（未来）
8. 忽略的单词（未来）

说明：

1. “常查的单词”来自搜索历史
2. 示例规则为“7 天内查询超过 3 次的词”
3. “忽略的单词”与“砍掉的单词”不同
4. “砍掉”表示掌握
5. “忽略”表示暂时不纳入学习流程

当前版本建议：

1. 先实现已有 6 类
2. 为“常查”和“忽略”保留卡片位或数据结构支持
3. 不强制在首版 UI 中展示未来类目

### 4.3.5 我的单词本

该区域复刻“单词块”的单词本管理逻辑，至少包括：

1. 单词本列表展示
2. 创建单词本
3. 查看单词本详情
4. 单词本增删改查

未来能力但本期不做：

1. 从照片/文件中识别导入
2. 单词册功能

### 4.3.6 卡片展示建议

根据截图，单词本卡片应至少包含：

1. 单词本名称
2. 单词数量
3. 视觉色块/进度色带
4. 更多操作入口

PPT 指出颜色会受“总词数 / 已掌握词数”的比值影响，因此卡片颜色不是纯静态皮肤，后续应支持根据进度映射。

## 4.4 单词本详情页 `/wdBook`

### 4.4.1 页面目标

展示某个单词本内的单词网格，并支持按不同规则排序浏览。

### 4.4.2 页面结构

从截图可见：

1. 顶部栏显示当前单词本名称
2. 顶部提供搜索入口
3. 主体为单词方块矩阵

### 4.4.3 排序要求

必须预留两种排序方式：

1. 按添加顺序
2. 按字母顺序

PPT 指出参考 App 默认是按字母排序，但本产品要额外支持按添加顺序。

### 4.4.4 交互要求

1. 排序模式在左上角功能按钮中切换
2. 左上角按钮弹出的其他功能暂不实现

### 4.4.5 数据要求

每个 `book` 需要维持两个顺序视图：

1. 添加顺序数组
2. 字母顺序数组

单词本单词上限为 2000 个。

### 4.4.6 颜色语义

PPT 和截图共同说明该页面用颜色表达学习状态，但当前只明确了部分规则：

1. 灰色：忽略
2. 橙色：应该学习
3. 紫色：存在特定语义，但未完全定义

建议将颜色映射做成配置表，而不是写死在组件内。

## 5. 数据模型需求

## 5.1 全局词典数据 `wordJson`

PPT 定义：

```json
{
  "go": {},
  "advance": {}
}
```

该结构本质是一个按单词 key 索引的总词库，负责存储基础词典信息。

应包含但不限于：

1. 音标
2. 词性
3. 释义
4. 查词页与页面标注所需的其它基础数据

数据来源：

1. 从服务器获取
2. 按需加载

说明：当前仓库 `src/data/dictionary/*.json` 很像单词级静态词典数据，后续可以评估是否作为 `wordJson` 的本地数据来源。

## 5.2 用户状态数据 `wordJsonU`

PPT 定义：

```json
{
  "go": {
    "s": "a/b/c/d",
    "a": "0~7",
    "sc": "searchCount",
    "t": "timestamp",
    "l": ["bookId"]
  },
  "advance": {}
}
```

推荐按单词为 key 保存，示意如下：

```json
{
  "go": {
    "s": "a",
    "a": 2,
    "sc": 5,
    "t": 1761955200000,
    "l": [1, 3]
  }
}
```

字段说明：

1. `s`：单词状态
2. `a`：艾宾浩斯学习阶段，仅 `s=a` 时有效
3. `sc`：查询次数
4. `t`：时间戳
5. `l`：所属单词本 id 列表

### 5.2.1 状态字段 `s`

PPT 明确定义：

1. `a`：正常背诵流程
2. `b`：ban，表示忽略，不进行学习和背诵
3. `c`：cut，表示砍掉，即掌握
4. `d`：ed，表示已背会

说明：

1. `b` 对应“忽略”
2. `c` 对应“砍掉的单词”
3. `d` 与 `c` 的边界目前并不完全清晰，建议产品补充定义

### 5.2.2 学习阶段字段 `a`

`a` 取值 `0~7`，对应间隔为：

1. `0`：刚添加
2. `1`：20 分钟后
3. `2`：1 小时后
4. `3`：9 小时后
5. `4`：1 天后
6. `5`：2 天后
7. `6`：6 天后
8. `7`：31 天后

### 5.2.3 时间戳字段 `t`

PPT 特别强调 `t` 逻辑最复杂。

规则应为：

1. 当 `s=b` 或 `s=c` 时，`t` 记录状态变更时间
2. 当 `s=a` 时，`t` 表示“下次应复习时间”
3. 该时间由“当前时间 + 阶段延迟”计算得出

这意味着 `t` 不是统一语义字段，而是“状态相关时间字段”。实现时要在文档或代码注释中明确，否则后续容易误用。

## 5.3 搜索历史 `searchList`

PPT 定义：

```json
["word0", "word1"]
```

规则：

1. 按时间从新到旧
2. 去重

建议：

1. 仅保存单词顺序
2. 查询次数放到 `wordJsonU[word].sc`
3. 展示时再从 `wordJson` 读取释义

## 5.4 待复习列表 `studyList`

PPT 定义：

```json
["word0", "word1"]
```

规则：

1. 按时间从新到旧
2. 去重

作用：

1. 作为“待复习单词”或单词块学习流的快速索引
2. 用于驱动单词块首页中的某些分类集合

## 5.5 单词本索引 `wdBookList`

PPT 定义：

```json
[id0, id1, id2]
```

规则：

1. id 可使用数字自增
2. 主要用于排序展示
3. 数组长度通常不会超过 20，极限不超过 100

## 5.6 单词本详情字典 `wdBookJson`

PPT 给了两类对象：

1. `book`
2. `list`

建议结构如下：

```json
{
  "1": {
    "s": "b",
    "n": "雅思核心词",
    "r": [50, 13],
    "wordsByAdd": ["go", "advance"],
    "wordsByAlpha": ["advance", "go"]
  },
  "2": {
    "s": "l",
    "n": "英语课本",
    "l": [1, 3, 5]
  }
}
```

字段含义：

1. `s=b`：book
2. `s=l`：list，表示单词册/单词本集合
3. `n`：名称
4. `r:[total, mastered]`：总词数和已掌握词数
5. `l`：该集合下包含的 book id

PPT 原文没有给出 `book` 内部保存单词数组的字段名，但由于第 5 页明确要求两种排序都要支持，因此实现上建议补充：

1. `wordsByAdd`
2. `wordsByAlpha`

否则无法稳定支撑排序切换。

## 5.7 变更数据集 `upDateList`

PPT 提到了：

```json
[]
```

名称为 `upDateList`，说明是“变更数据集”。

PPT 没有解释其最终用途，但结合场景，可能是：

1. 用户状态变更待同步队列
2. 本地数据脏标记集合
3. 学习状态批量更新日志

当前建议：

1. 先预留结构
2. 不作为首版必须实现项
3. 若有离线同步需求，再细化其数据格式

## 6. 业务规则整理

## 6.1 搜索行为

当用户查询一个单词时，至少应执行：

1. 若 `wordJson` 中无该词，触发远程加载
2. 更新 `searchList`，置顶并去重
3. `wordJsonU[word].sc + 1`
4. 记录最近查询相关时间

## 6.2 单词加入学习流

当用户把词加入学习流时：

1. `wordJsonU[word].s = a`
2. `wordJsonU[word].a = 0`
3. 计算 `t = now`
4. 同步更新 `studyList`
5. 同步更新所属 `book` 的单词数组

## 6.3 忽略

当用户忽略一个词时：

1. `wordJsonU[word].s = b`
2. `wordJsonU[word].t = 状态变更时间`
3. 该词不再参与正常学习/背诵流程

## 6.4 砍掉或掌握

当用户判定已掌握时：

1. `wordJsonU[word].s = c` 或 `d`
2. `wordJsonU[word].t = 状态变更时间`
3. 同步更新单词本统计数据 `r`

这里最大的产品歧义是 `c=cut` 和 `d=ed` 的区别。当前建议在正式开发前先统一定义，否则会导致分类统计和颜色映射冲突。

## 6.5 常查的单词

未来规则示例：

1. 7 天内查询次数超过 3 次

建议不要把这个类目独立存储为永久列表，而是根据 `wordJsonU[word].sc` 和时间窗口动态计算或周期计算。

## 7. 建议的 MVP 范围

### 7.2 功能

## 8. 技术实现建议

## 8.1 数据分层

建议分成三层：

1. 词典基础层：`wordJson`
2. 用户状态层：`wordJsonU`
3. 视图索引层：`searchList`、`studyList`、`wdBookList`、`wdBookJson`

这样做的好处是：

1. 基础词典数据与用户行为解耦
2. 视图排序可独立优化
3. 后续远程同步或本地缓存更容易扩展

## 8.2 不要重复存释义

PPT 已明确表达“历史数组只需要单词即可”，因此：

1. `searchList`
2. `studyList`
3. `book` 内单词数组

都尽量只存单词 key，不重复存释义、音标等详情。

## 8.3 排序能力要在数据层准备好

因为单词本详情页必须支持两种排序，所以建议不要在每次渲染时都临时排序大数组，而是预先维护：

1. `wordsByAdd`
2. `wordsByAlpha`

这样切换排序只是切换数据源引用。

## 8.4 颜色映射配置化

颜色语义目前还不完整，尤其紫色未定义清楚。建议设计：

```ts
type WordStatusColorMap = {
  ignore: string
  learn: string
  review: string
  mastered: string
}
```

先把颜色策略做成配置，避免后续改状态含义时需要大量改 UI 代码。

## 9. 待确认问题

以下问题在 PPT 中存在歧义，正式开发前最好先确认：

1. 首页 3 个 Tab 是页面内切换，还是“单词块”直接跳转到模块页。
2. 查词详情页除了头部两行，正文部分是否需要做完整释义折叠区。
3. `searchList` 是否真的只存单词，查询次数是否统一落到 `wordJsonU.sc`。
4. `c=cut` 与 `d=ed` 的业务区别是什么。
5. 紫色在单词块页中的确切语义是什么。
6. AI 单词本当前首版到底展示 6 类还是 8 类。
7. `studyList` 与“待复习单词”分类的关系是完全等价还是部分交集。
8. `upDateList` 是否需要在首版就承担离线同步职责。
9. 单词本详情页的左上角菜单除排序外，未来还会包含哪些操作。
10. 单词本与单词册是否都要在首版的数据结构中一次性支持。

## 10. 推荐开发顺序

### 第一阶段：数据底座

1. 定义 `wordJson`
2. 定义 `wordJsonU`
3. 定义 `searchList`
4. 定义 `studyList`
5. 定义 `wdBookList` / `wdBookJson`

### 第二阶段：基础页面

1. 首页搜索与历史
2. 查词详情页
3. 单词块首页静态布局
4. 单词本详情页静态布局

### 第三阶段：业务联动

1. 查词后更新历史与计数
2. 加入学习流
3. 单词状态变更
4. 单词本统计联动
5. 排序切换

### 第四阶段：增强功能

1. 常查词计算
2. 忽略词分类
3. 顶部统计模块真实数据
4. OCR/文件导入
5. 单词册

## 11. 结论

这份 PPT 的本质不是单一页面需求，而是一套“词典查询 + 生词管理 + 单词本学习”的产品草案。

从开发角度，当前最重要的是先把以下三件事做对：

1. 页面范围收敛，只做 MVP
2. 数据结构先定清楚，尤其 `wordJsonU`、`searchList`、`wdBookJson`
3. 把“当前必做”和“未来预留”严格分开

只要先完成这三点，后续无论是接真实词典数据、补统计模块，还是扩展 OCR 和单词册，都不会推翻首版结构。
